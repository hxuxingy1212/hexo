HashMap

底层数据结构，1.7与1.8 有何不同？

  • 1.7数组＋链表
  • 1.8数组+（链表|红黑树）

如何解决链表长度过长的？

  • 扩容（有局限性，当原始hash一样时，即使扩容，位置不变,所以链表长度是可以超过8的，但一般场景下，链表长度可以减半）
  • 树化 (没有达到树化条件不会树化，而是选择扩容来解决)

为何要用红黑树

  • 红黑树解决了链表过长查询性能较低问题
  • 当原始hash一样时，即使扩容也不会减少链表长度，这时需要树化来解决。
  • 防止dos攻击，不断添加hash相同的元素，导致链表超长，此时树化可以防止

为何一上来不树化

  • 性能：短链表性能不比红黑树差
  • 内存：红黑树结点对象更复杂，占用更高内存

树化國值为何是8
红黑树是偶尔情况，主要是防止dos攻击的，一般情况不会树化，hash 值如果足够随机，则在 hash 表内按泊松分布，在负载因子 0.75的情况下，长度超过8的链表出现概率是0.00000006，选择8就是为了让树化几率足够小

何时会树化，何时会退化为链表？

  • 树化两个条件：链表长度超过树化阈值(8)；数组容量 >=64
  • 退化情况1：在扩容时如果拆分树时，树元素个数<=6则会退化链表，
  • 退化情况2：remove 树节点时，若 root、 root.left. root.right. root.left.left 有一个为 null，也会退化为链表,移除这些结点时不会，移出后，再移出其他任意树结点则会。

索引如何计算？

  • 首先调用hashcode方法计算原始hash，再调用hash方法计算出二次hash，与数组容量取模，就是索引下标（桶下标）

hashCode都有了，为何还要提供 hash() 方法？

  • 二次 hash() 是为了综合高位数据，让哈希分布更为均匀

数组容量为何是2的n次幂？

  • 取索引时取模运算例如 97%16=1 可以等价于 97 &（16-1），计算效率提升
  • 扩容时，拿二次hash与原始容量按位与，为0则不需要移动，不为0则需要移动，移动位置为index + 原始容量
    注意：java中hashMap的这一实现是为了注重性能，如果元素都是偶数，那么奇数索引都不可能有元素，分布不均，如果注重分布，容量应该为质数，二次扩容，取模优化，扩容优化都是在这一设计上实现的来提高效率的。

何时扩容？

  • 1.7:添加某个元素后，数组的总的添加元素数大于 数组长度 * 0.75(默认,也可自己设定) 并且 没有空位时才扩容。
  • 1.8:添加某个元素后，数组的总的添加元素数大于 阈值,数组长度扩容为两倍

注意

  • 1.7扩容机制更合理
  • 先添加元素在原数组 再扩容

介绍一下 put 方法流程，1.7与1.8 有何不同？
流程
HashMap 是懒情创建数组的，首次使用才创建数组
计算索引（桶下标）
如果桶下标还没人占用，创建 Node 占位返回
如果桶下标已经有人占用

  • 已经是 TreeNode 走红黑树的添加或更新逻辑
  • 是普通 Node，走链表的添加或更新逻辑，达到树化条件，走树化逻辑
    返回前检查容量是否超过阈值，一旦超过进行扩容

不同

  • 键表插入节点时，1.7 是头插法，1.8 是尾插法
  • 1.7 是大于等于阈值且没有空位时才扩容，而1.8 是大于阈值就扩容
  • 1.8 在扩容计算 Node 索引时，会优化 , 1.7没有优化

加载因子为何默认是 0.75f?

  • 在空间占用与查询时间之间取得较好的权衡
  • 大于这个值，空间节省了，但链表就会比较长影响性能
  • 小于这个值，冲突减少了，但扩容就会更频繁，空间占用多

多线程下会有啥问题？

  • 扩容死链（1.7）
  • 数据错乱（1.7，1.8）
    数据错乱: 多线程下，添加2个key 他们索引位置一样，因为执行时间的问题，后者会覆盖前者，导致前者添加失败，数据丢失
    扩容死链: 多线程下，同时扩容，一边先扩容完，头插法导致顺序颠倒 ，另一边 继续进行扩容流程 ，导致后扩容的线程 ，链表发生了闭环，当查询时，无限循环，cpu空转。

key 能否为 null，作为 key 的对象有什么要求？

  • HashMap 的 key 可以为 null，但 Map 的其他实现则不然
  • 作为 key 的对象，必须实现 hashCode 和 equals，并且 key 的内容不能修改（不可变）

String 对象的 hashCode(） 如何设计的，为啥每次乘的是31
目标是达到较为均匀的散列效果，每个字符串的 hashCode 足够独特
字符串中的每个字符都可以表现为一个数字，称为Si，其中i的范围是0~n-1
散列公式为：S031”-1+ S131”-2+ .S * 31”-1- + . Sn-1* 319
31 代入公式有较好的散列特性，并且31h 可以被优化为
即32h -h
即25 *h -h
即h《5 -h

ConcurrentHashMap

对比

  • Hashtable 与 ConcurrentHashMap 都是线程安全的Map 集合
  • key和vule都不可以为null
  • Hashtable 并发度低，整个 Hashtable 对应一把锁，同一时刻，只能有一个线程操作它
  • 1.8 之前 ConcurrentHashMap 使用了 Segment +数组 +链表的结构，每个 Segment 对应一把锁，如果多个线程访问不同的Segment，则不会冲突
  • 1.8 开始 ConcurrentHashMap 将数组的每个头节点作为锁，如果多个线程访问的头节点不同，则不会冲突

hashtable

  • 初始容量为11，每次扩容*2+1
  • 不需要2次hash
  • 线程安全性：在hashmap的基础上所有方法使用synchronized修饰

V1.7

容器中有多把锁，每一把锁锁一段数据，这样在多线程访问时不同段的数据时，就不会存在锁竞争了，这 样便可以有效地提高并发效率。

初始化
构造函数参数：capacity ，factor clevel
容量/并发度 = 每个segment内数组容量，但最小不小于2
饿汉式，具体创建看下面segment[0]原型

索引计算
2次hash后，高n位 就是 segment位置，低n位，就时segment内数组桶的位置
扩容
segment不扩容，初始化时就固定了，扩容的只是segment内的小数组，且独自扩容，扩容规则和hashmap一样
segment[0]原型
初始化时，只会创建segment[0],其他的segment会在添加元素时复制segment[0]的原型来创建
segment[0]原型扩容了，后面按照segment[0]来创建的也会和segment[0]一样大小

V1.8

初始化
懒汉式，第一次添加元素时才创建
理解capacity ，factor
capacity不是初始化容量了，而是预计要放多少个元素，会更具这个值和factor来调整。
例如，capatity=11 factor为0.5 第一次put时初始容量为32
factor只在第一次创建时有效，后面按照0.75的扩容因子扩容
并发put
锁的只是链表头，当访问不同桶下标时，不会出现并发安全问题，也不会上锁
当访问桶下标相同时，会被锁阻塞，依次执行
扩容
扩容时从后往前扩容，扩容后会标记为forwardingNode，表示已经被扩容移到了新数组中

扩容get

  • 当get被forwardingNode标记的元素，在新数组中找，
  • 当get正在正在扩容的结点时,再旧数组中找。
  • 当get没有被标记的结点时，会并发的在老数组中获取结点的元素

注意：扩容链表时，会复制一份不同的新的结点，防止结点指向后续结点不同，导致数据错乱,防止扩容get时get到的对象不一样
扩容put

  • put被forwardingNode标记的元素，会帮助扩容线程扩容，扩容完成再put
  • put正在迁移的前半部分，CAS加synchronized头结点来保证线程安全
  • put正在迁移的部分，重试，等迁移当前结点后，帮助迁移再put

ThreadLocal

谈一谈对 ThreadLocal 的理解

  • ThreadLocal可以实现【资源对象】的线程隔离，让每个线程各用各的【资源对象】，避免争用引发的线程安全问题
  • ThreadLocal 同时实现了线程内的资源共享

ThreadLocal与ThreadLocalMap关系
其原理是，每个线程内有一个 ThreadLocalMap 类型的成员变量，用来存储资源对象

  • 调用 set 方法，就是以 ThreadLocal 自己作为 key，资源对象作为 value，放入当前线程的 ThreadLocalMap 集合中
  • 调用 get 方法，就是以 ThreadLocal 自己作为 key，到当前线程中查找关联的资源值
  • 调用 remove 方法，就是以 ThreadLocal 自己作为 key，移除当前线程关联的资源值

hash值怎么计算
第一次是0，后面每次累计一个数字作为hash值

扩容
初始16，当超过2/3时扩容

hash冲突怎么解决
冲突时，不断找下一个，直到这个位置没有元素

为什么key设置为弱引用
线程有时需要长时间运行，比如线程池中的线程，因为担心程序员没有良好的编程习惯，用完不回收，所以设置弱引用使得将来没有被引用时，被垃圾回收掉
但 GC 仅是让 key 的内存释放，后续还要根据 key 是否为 null 来进一步释放值的内存，释放时机有

  • 获取 key 发现 null key , 会使value为null ，并且添加一个key在该位置上
  • set key 时，会使用启发式扫描，清除临近的 null key，启发次数与元素个数和是否发现 null key 有关
  • 主动 remove 时（推荐），因为一般使用 ThreadLocal 时都把它作为静态变量，因此 GC 无法回收
    注意：ThreadLocal的put与其他map不一样，当发现key为null时，将value设置为null ， 并将key设置
